## TL;DR
Dynamic web applications are usually split into **three tiers**, presentation (client), application/middleware (server), and data (database). When you type a URL, the browser walks a sequence of network steps (DNS lookup, TCP handshake, HTTP request, server response, render) to reach the right server and pull a resource. The restaurant analogy (dining area → kitchen → cold-storage) is a quick mental model for the same split.

## Web applications, static vs dynamic ([[web-application]])

| Aspect | Static | Dynamic |
|---|---|---|
| User interaction | None or minimal | Input, change, manipulate data |
| Content source | Files on disk | Generated by server programs, often from a [[database]] |
| Example | Yahoo Weather (historically) | Amazon, social networks, online editors |
| Behaviour | Same response for everyone | Response personalised per user/request |

A **web application** is a client-server program where the UI runs in a browser. Distinguish from a **desktop application**: web apps run on a server and are reached via HTTP; desktop apps run locally via the OS. Trade-off: web apps give multi-device access at the cost of speed and limited local resources.

## Life and times of a web request ([[http-request-lifecycle]])

When the user types a URL:

1. Browser parses the URL.
2. **Browser cache** checked.
3. **DNS lookup**, domain name resolved to an IP. DNS = "internet's phone book". See [[dns]].
4. **TCP connection** opened to the server (transmission control protocol). See [[tcp]].
5. Browser sends an **HTTP request** for the resource (hypertext transfer protocol). See [[http]].
6. Server handles the request, building or locating the resource.
7. Browser receives the **HTTP response**.
8. Browser parses the HTML and renders the page.

For a **dynamic** site, step 6 is more involved: the server runs a program (often querying a database) to generate the response on the fly rather than reading a static file.

## Three-tier architecture ([[three-tier-architecture]])

```
[ Presentation ]    Client side    HTML, CSS, JavaScript (browser)
       |
       | HTTP
       v
[ Application  ]    Server side    JavaScript (Node), PHP, Python, Ruby
       |
       | SQL
       v
[    Data      ]    Server side    Relational DB (e.g. MySQL), file systems
```

- **Presentation / front end**: receives user input, displays output.
- **Application / middleware**: business logic, calculations, request handling. Centralises logic between client and data. This intermediate layer is what separates three-tier from older **two-tier** designs (in which the client talks straight to the DB).
- **Data / back end**: persistence, storage, management.

Why three-tier beats two-tier (Ceri et al., *Designing Data-Intensive Web Applications*):
- Better **network utilisation**.
- Middle tier can be **replicated and load-balanced**, giving scalability the two-tier model can't.
- Cleaner [[separation-of-concerns]], each layer evolves independently.

### Client vs server side
| Side | Tiers | Typical languages |
|---|---|---|
| Client | Presentation | HTML, CSS, JavaScript |
| Server | Application + Data | JavaScript (Node), PHP, Python, Ruby, SQL |

## Restaurant analogy
| Restaurant | Three-tier web app | Side |
|---|---|---|
| Dining area, tables, menus, customers see | Presentation tier | Client |
| Kitchen, chefs, where requests are handled | Application tier | Server |
| Fridge, freezer, pantry, ingredient store | Data tier | Server |

Useful when you can't remember which tier does what: "is this thing the customer sees, or the kitchen, or the pantry?"

## Common pitfalls / exam fodder
- Confusing **tier** with **layer** physically: tiers are logical, can run on the same machine or many machines.
- Mixing up which languages live where (e.g. SQL is a data-tier language, not client side).
- Forgetting the difference between **TCP** (transport, opens the connection) and **HTTP** (application-level, asks for resources). HTTP runs on top of TCP.
- Calling something "dynamic" purely because it has animations: dynamic = content generated per request, usually data-driven.
